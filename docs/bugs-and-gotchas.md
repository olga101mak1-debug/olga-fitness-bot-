# Bugs and gotchas

Этот файл нужен, чтобы не наступать на одни и те же ошибки дважды.

## Как добавлять запись

Каждая запись должна отвечать на 5 вопросов:

1. Что сломалось?
2. Как это выглядело?
3. Почему это произошло?
4. Как исправили?
5. Как проверить, что больше не сломано?

## Записи

### 2026-07-23 — Пустой ответ Claude падал в Telegram с общей заглушкой

- Симптом: бот показывал «Что-то пошло не так» вместо содержательного ответа на сложные аналитические запросы (например, сравнение показателей за несколько дат).
- Причина: `claude_client.py` при пустом ответе от Claude (`response.content` без text-блока) пытался отправить пустую строку в Telegram → `TelegramBadRequest: message text is empty` → падал в общий catch.
- Решение: добавлен retry на пустой ответ + честное сообщение об ошибке вместо тихого падения.
- Проверка: диагностический скрипт, который вызывает реальный Anthropic API с реальным контекстом из БД и печатает `stop_reason`/`usage`/содержимое `response.content`.
- Где связано: запись ниже (корневая причина)

### 2026-07-23 — Корневая причина пустых ответов: max_tokens слишком мал

- Симптом: на сложных аналитических запросах модель тратила весь бюджет токенов на `thinking` и не успевала написать текст (`stop_reason: max_tokens`, 0 текстовых блоков).
- Причина: `max_tokens=500` в `coach.py` и `weekly_analyst.py` был слишком мал для такого типа запросов.
- Решение: подняли `max_tokens` до 2000 в обоих местах — **но только на сервере**, в этом GitHub-репозитории всё ещё старое значение (проверено 23.07.2026, см. `project-memory.md` → «⚠️ Важно»).
- Проверка: `grep max_tokens app/services/ai/coach.py app/services/ai/weekly_analyst.py` — если видно 500/400, значит фикс на сервере, но не в репозитории.
- Где связано: `project-memory.md` → «⚠️ Важно: деплой вручную по SFTP, не через git»

### 2026-07-23 — Бот врал, что обхватов тела нет, хотя они были в БД

- Симптом: на вопрос про изменения обхватов бот уверенно отвечал, что таких замеров нет.
- Причина: `_format_context()` в `coach.py` собирал тренд по последним 14 дням только с полями `weight/mood/sleep_hours/work_hours/stress` — обхваты (`waist/belly/hips/neck/chest`) в контекст не попадали вообще, хотя были записаны в БД.
- Решение: добавили эти поля в `trend`.
- Проверка: спросить бота про динамику обхватов при наличии записанных данных — ответ должен использовать реальные цифры, а не отрицать их наличие.
- Где связано: `project-memory.md` → «Долговечные факты»
